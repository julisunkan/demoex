================================================================================
  MAILVAULT PRO — COMPLETE DEPLOYMENT GUIDE
  Self-Hosted Outlook Add-in · USDT Payments & License Keys
================================================================================

Last updated: July 2026
This guide walks you through deploying MailVault Pro to production and
configuring it for your organization. There is no Microsoft AppSource
listing, no Azure AD app registration, and no Microsoft sign-in required —
MailVault Pro is fully self-hosted and self-licensed.

Read this entire document before starting. Steps must be done in order
because later steps depend on values you collect in earlier ones.

--------------------------------------------------------------------------------
PART 0 — WHAT YOU ARE DEPLOYING
--------------------------------------------------------------------------------

MailVault Pro is a Microsoft Outlook add-in with two parts:

  1. Frontend (React + Vite)
     A task-pane web app that runs inside Outlook. It shows the backup,
     restore, cleanup, and analytics dashboards, plus a Settings page where
     users unlock Pro via a USDT payment or a license key.

  2. Backend (Node.js / Express)
     An API server that handles:
       - Mailbox backup, restore, and cleanup operations
       - Analytics data
       - Billing: USDT payment submissions and license key issuance/redemption
       - Admin settings, license management, and payment approval
       - Optional outgoing notification webhooks and email reminders

Billing works as follows:
  - Users install the add-in for free and get the Free tier immediately —
    no account, no Microsoft sign-in.
  - To unlock Pro, a user either:
      a) Sends a USDT payment to your wallet address and submits the
         transaction hash in the app. An admin reviews and approves it in
         the admin portal, which automatically issues a license key.
      b) Enters a license key you issued directly (e.g. for enterprise
         deals, trials, or manual sales channels).
  - You control 100% of the payment and licensing flow — there is no
    third-party marketplace or billing intermediary.

Current pricing plans (configured in server/data/settings.json, editable
from the admin portal's Billing Settings panel):
  • Monthly:  $9.99  / month
  • Annual:   $99.99 / year (17% discount)

Free features (no license needed): 1 mailbox, local storage only, manual
backups, basic analytics, community support.

Pro features (USDT payment or license key required): unlimited mailboxes,
all cloud storage providers (Local, Azure Blob, OneDrive, Amazon S3, Google
Drive, Dropbox), AES-256-GCM encryption, automated scheduling, AI-powered
analytics, multi-user/group support, custom retention policies, advanced
audit logs, and priority support.

PART 1 — WHAT YOU NEED BEFORE STARTING
--------------------------------------------------------------------------------

  A) A hosting environment for the Node.js app
     → Replit Deployments is the simplest path (this project is already
       configured for it — see Part 2).
     → Any other Node.js host (Render, Railway, Fly.io, a VPS, etc.) also
       works since the app has no platform-specific dependencies.

  B) A custom domain (recommended for production)
     → Required so Outlook can load your add-in over HTTPS from a stable
       URL. Replit Deployments gives you a `*.replit.app` HTTPS domain out
       of the box; you can attach a custom domain later if you want one.

  C) A USDT wallet address (for accepting crypto payments)
     → Any TRC-20 (Tron network) or ERC-20 (Ethereum network) USDT wallet
       you control. You will enter this address in the admin portal —
       no third-party payment processor is involved.

  D) (Optional) An SMTP account for email notifications
     → Only needed if you want the app to email you when a payment is
       submitted, or email users when their license is about to expire.
     → Any standard SMTP provider works (Gmail, SendGrid, Mailgun, etc.).

  E) (Optional) A Microsoft 365 tenant with admin access
     → Only needed if you plan to push the add-in to your whole
       organization via Centralized Deployment (Part 5). Not required for
       individual sideloading/testing.

PART 2 — DEPLOY THE APPLICATION
--------------------------------------------------------------------------------

STEP 2.1 — Deploy on Replit (recommended — fastest path)
  This project is already configured for Replit Deployments:
    - Build command:  npm run build
    - Run command:    node ./dist/index.cjs
    - Public dir:     dist/public
    - Deployment target: autoscale

  → Open the Deployments tab in your Replit workspace
  → Click "Deploy"
  → Set the required environment variables (see Part 3) as Secrets before
    or during setup
  → Once deployed, note your public URL, e.g. https://your-app.replit.app
    This is the domain you will use everywhere below as YOUR_DOMAIN.

STEP 2.2 — Deploy elsewhere (alternative)
  If you prefer another host, the app is a standard Node.js app:
    → Build:   npm run build
    → Start:   node ./dist/index.cjs   (serves both the API and the
               built frontend from a single Node process, listening on
               the PORT env var)
    → Point your host's build/run commands at the above.
  Make sure your host serves the app over HTTPS on a stable domain —
  Outlook will not load an add-in manifest served over plain HTTP.

STEP 2.3 — Update the manifest with your real domain
  Open manifest.xml in this project and replace every occurrence of
  YOUR_DOMAIN with your actual deployed domain (e.g.
  your-app.replit.app). This includes:
    - IconUrl, HighResolutionIconUrl
    - SupportUrl
    - AppDomains → AppDomain
    - FormSettings → DesktopSettings → SourceLocation
  Also generate a fresh, unique GUID for the <Id> field (use
  https://www.guidgenerator.com) — do not ship the placeholder ID.

  Your manifest will be served automatically at:
        https://YOUR_DOMAIN/manifest.xml
  You can also view/download a live copy from inside the app at
  Settings → Deployment Guide → Manifest Reference.

PART 3 — ENVIRONMENT VARIABLES
--------------------------------------------------------------------------------

Environment variables store secrets and configuration outside your code.
Set these as Replit Secrets (or your host's equivalent). Never commit
secrets to Git.

REQUIRED
========

  ADMIN_PASSWORD
    What it is: A password you choose to protect the /admin panel and its
                API routes.
    Why needed: The admin portal lets you configure billing settings,
                approve USDT payments, and issue/revoke license keys. This
                password gates every admin API request (sent as the
                x-admin-password header).
    Recommendation: Use a password manager to generate a 20+ character
                random password. If left unset, the admin portal is open
                to anyone — never leave this unset in production.

  VITE_ADMIN_PASSWORD
    What it is: The same password as ADMIN_PASSWORD, exposed to the
                frontend build so the admin login screen can send it.
    Why needed: Must match ADMIN_PASSWORD exactly, or admin login will
                fail with 401s.

OPTIONAL
========

  PORT
    What it is: The port the server listens on.
    Default:    3001 (or 5000 in the Replit dev workflow)
    Note:       Most hosts, including Replit Deployments, set this
                automatically — you usually don't need to set it yourself.

  SMTP_HOST, SMTP_PORT, SMTP_USER, SMTP_PASS, SMTP_FROM
    What they are: Credentials for an SMTP server.
    Why needed: Only required if you enable email notifications (payment
                alerts, license expiry reminders) from the admin portal's
                Billing Settings panel. Leave blank to disable email
                features entirely — the app works fully without them.

There is no AZURE_TENANT_ID, AZURE_CLIENT_ID, AZURE_CLIENT_SECRET, or any
other Microsoft/Azure credential to configure. MailVault Pro does not use
Microsoft sign-in or the Microsoft SaaS Fulfillment API.

PART 4 — CONFIGURE BILLING IN THE ADMIN PORTAL
--------------------------------------------------------------------------------

Once deployed, log in to the admin portal at https://YOUR_DOMAIN/admin
using your ADMIN_PASSWORD.

STEP 4.1 — Set your USDT wallet address
  → Go to "Billing Settings"
  → Enter your USDT wallet address and select the network (Tron/TRC-20 or
    Ethereum/ERC-20)
  → This is the address users are shown when they choose to pay with USDT

STEP 4.2 — Review or edit pricing plans
  → Still in "Billing Settings", edit the Monthly and Annual plan prices,
    labels, and durations as needed
  → Changes take effect immediately — no redeploy required (plans are
    stored in server/data/settings.json)

STEP 4.3 — Approve USDT payments
  → Go to "USDT Payments"
  → When a user submits a transaction hash, review it against your wallet's
    transaction history
  → Click "Approve" to automatically issue a license key to the user, or
    "Reject" if the payment can't be verified

STEP 4.4 — Issue license keys manually
  → Go to "Licenses"
  → Click "Create License" to generate a key you can hand out directly
    (for enterprise deals, trials, resellers, etc.) without requiring a
    USDT payment
  → Keys can be revoked at any time from the same panel

STEP 4.5 — (Optional) Configure notifications
  → Enable email reminders for licenses nearing expiry, and/or set an
    outgoing webhook URL to receive payment.submitted, payment.approved,
    license.redeemed, and license.expiring events in your own systems
    (Slack, Zapier, a ticketing system, etc.)

PART 5 — CENTRALIZED DEPLOYMENT TO YOUR ORGANIZATION (OPTIONAL)
--------------------------------------------------------------------------------

If you want to push MailVault Pro to every mailbox in your Microsoft 365
tenant without any user action, use Centralized Deployment via the
Microsoft 365 Admin Center. No AppSource listing or app review is
required — you upload your own manifest directly.

STEP 5.1 — Sign in to the Microsoft 365 Admin Center
  → Go to https://admin.microsoft.com
  → Sign in with a Global Administrator account

STEP 5.2 — Upload the custom add-in
  → Go to Settings → Integrated apps
  → Click "Upload custom apps"
  → Choose "Office Add-in" and upload your manifest.xml, or provide the
    manifest URL: https://YOUR_DOMAIN/manifest.xml
  → Select which users or groups should receive it (or deploy to everyone)
  → Click "Deploy"

  This uploads your own manifest directly — no marketplace listing or
  Microsoft billing is involved. Users will see MailVault Pro appear in
  Outlook automatically within a few hours.

STEP 5.3 — Test before organization-wide rollout
  Sideload the manifest for yourself first:
    → In Outlook on the web, go to Get Add-ins → My add-ins →
      Add a custom add-in → Add from URL / Add from file
    → Provide https://YOUR_DOMAIN/manifest.xml (or the local file)
  Confirm the add-in loads, backups/restores work, and the Settings page
  correctly shows Free vs Pro state before deploying to your whole
  organization.

--------------------------------------------------------------------------------
PART 6 — TESTING THE BILLING FLOW BEFORE GOING LIVE
--------------------------------------------------------------------------------

STEP 6.1 — Verify the backend is healthy
  → Open: https://YOUR_DOMAIN/api/health
  → You should see: {"ok": true, "service": "MailVault Pro API", ...}

STEP 6.2 — Test license issuance and redemption
  → Log in to /admin and create a test license key
  → In the app's Settings page, enter the key under "Have a license key?"
  → Confirm the app switches to Pro state immediately

STEP 6.3 — Test the USDT payment flow end-to-end
  → In the app, choose "Pay with USDT", copy the wallet address shown,
    and submit a test transaction hash
  → In /admin → USDT Payments, approve the submission
  → Confirm a license key is auto-issued and the app unlocks Pro

STEP 6.4 — Confirm no Microsoft sign-in is required anywhere
  → The app should never prompt for a Microsoft account. If it does,
    you're running an outdated build — redeploy from the current codebase.
